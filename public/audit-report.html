2. Broken / Incomplete Workflows

Build/Revision Synchronous Execution

AI builds run synchronously inside the request handler. Cloudflare Workers free tier has a 10ms CPU limit per request. GPT-4o calls easily exceed this in production, causing silent timeouts. The build_jobs table exists for async tracking but the actual AI call never offloads to a queue.

CRITICAL — Will fail silently in production
Deployment is Fully Simulated

simulateDeployment() sets status to 'live' after a setTimeout(2000). No Cloudflare Pages API call is made. The deployment_url is a made-up .pages.dev URL that doesn't exist. Users are charged 15 coins for a non-deployment.

CRITICAL — Core promise is undelivered
Monthly Coin Grant — No Scheduler

The POST /api/vault/grant endpoint exists but must be called manually. There's no Cron Trigger or scheduled Worker to fire monthly grants. Free users never receive their 50 coins after the welcome bonus unless they manually call this endpoint.

HIGH — Revenue & retention impact
Subscription Cancellation Not Fully Handled

When customer.subscription.deleted fires, the membership record is updated but the cancellation date, grace period end, and coin revocation are not tracked. Users could retain Pro features after cancellation if the webhook fires late or is retried.

HIGH — Financial integrity risk
Password Reset Token Expiry — No Cleanup

Reset tokens are generated and stored in password_resets table. The table has an expires_at column but no cleanup job runs. Over time, stale tokens accumulate. The health-check cleans rate_limits but not password_resets.

MEDIUM — Security hygiene
Project Archive — No Cascade

Archiving a project sets status='archived' but doesn't cancel active coin_holds, mark build_jobs as cancelled, or remove deployments from the live count. Deployment limits can be artificially exhausted by archived projects.

MEDIUM — Data integrity
navigateTo Dual Declaration (Fixed)

Two function navigateTo() declarations existed in app.js. The second hoisted definition created an infinite recursion through the vault-balance hook. Fixed in this session — a safe inline hook was wired instead.

FIXED ✅
revenue-summary Column Mismatch (Fixed)

Query referenced be.coins_granted and be.package_id which don't exist in billing_events. Caused 500 errors on every call. Fixed in this session — query now uses actual columns from schema.

FIXED ✅

Build Job Flow (Phase 1 Target)
1.User POSTs /api/projects/:id/build
2.Worker validates coins, creates build_job, holds coins → D1
3.Worker sends message to BUILD_QUEUE with {jobId, userId, projectId, modelId}
4.API returns 202 immediately with {job_id, status: "queued"}
5.Queue consumer Worker wakes, calls AIService.processIntent()
6.AI response stored in generated_specs + generated_outputs → D1
7.build_job status → completed; coin hold released → D1
8.IntentLog.log() stores audit record with SHA-256 hash → D1
9.Notification created; Resend email sent to user
10.Frontend poll detects status=completed, renders output
Stripe Payment Flow
1.User selects coin package in Vault
2.POST /api/vault/checkout → StripeService.createCoinCheckoutSession()
3.Stripe API returns session URL → frontend redirects
4.User completes payment on Stripe-hosted page
5.Stripe sends checkout.session.completed webhook
6.Worker verifies HMAC signature (STRIPE_WEBHOOK_SECRET)
7.CoinService.credit() adds coins → coin_wallets + ledger → D1
8.billing_event recorded with external_id for idempotency
9.Resend email: "X coins added to your wallet"
10.Frontend redirect lands on Vault page with updated balance

R-01 · AI Build Timeout in Production
Impact: Critical · Likelihood: Certain if not addressed
Phase 1 Must-Fix

AI completions (gpt-4o) take 5–30 seconds. Cloudflare Workers free tier: 10ms CPU / request. Paid Workers: 30ms CPU. Both will timeout. Current synchronous approach will fail for all but the fastest gpt-4o-mini responses in production.

Mitigation
  • → Migrate to Cloudflare Queues (P1-1) — consumer has 15-min wall time
  • → Implement polling endpoint GET /api/projects/:id/build-status
  • → Health-check already auto-releases stuck jobs after 10 min ✅
Contingency
  • → If Queues not available: use streaming response + KV status updates
  • → Restrict prod to gpt-4o-mini only until async queue is live
R-02 · Coin Double-Spend on Network Retry
Impact: Critical · Likelihood: Medium
Phase 1

If a build request times out on the client side, the user retries. The first request may have already created a build_job and deducted coins (or placed a hold). A second request creates another hold. No idempotency key check exists on the build endpoint.

Mitigation
  • → Accept X-Idempotency-Key header on all coin-debit endpoints
  • → Store key in D1 for 24h; return cached response on duplicate
  • → Check for existing queued/processing job before creating new one
DB Change
  • → Add idempotency_key TEXT UNIQUE to build_jobs
  • → New table: idempotency_cache (key, response_json, expires_at)
R-03 · Stripe Secret Key Exposure
Impact: Critical · Likelihood: Low (if practices followed)
Ongoing

STRIPE_SECRET_KEY is currently in .dev.vars (local dev only — correct). The risk is accidental commit of .dev.vars or .env to a public repo, or key accidentally logged in console.error() calls that appear in Cloudflare Worker logs.

Mitigation
  • → .dev.vars already in .gitignore ✅
  • → Add pre-commit hook to scan for sk_live_ patterns
  • → Use wrangler secret put for all production keys
  • → Audit all console.error() calls — never log full env objects
Immediate Action
  • → Rotate STRIPE_PUBLISHABLE_KEY seen in live test response — it's currently in plaintext in API response which is acceptable, but ensure it's the intended live key
  • → Enable Stripe radar rules for unusual activity
🟡 High
R-04 · D1 Write Throughput Limits
Impact: High · Likelihood: Medium at scale
Phase 2 Monitor

D1 free tier: 100k writes/day, 5M reads/day. Paid: 25B reads, 50M writes/month. Each build creates ~8 DB writes. At 1,000 builds/day = 8,000 writes. Intent log + analytics writes add further. Could hit limits at moderate scale.

Mitigation:

Batch coin_analytics writes; use KV for high-frequency counters; add D1 metrics monitoring in admin dashboard; upgrade to D1 paid tier early ($0.001 per 1M writes).

R-05 · AI Provider Outage / Rate Limits
Impact: High · Likelihood: Medium
Ongoing

OpenAI or Anthropic outages will halt all AI-powered features. Current fallback logic exists in AIService but falls back to a demo response — users won't know their build was mocked.

Mitigation:

Mark demo responses clearly in the UI. Store fallback flag in generated_specs. Don't debit coins for demo responses. Add AI health check endpoint. Consider adding a third provider (Gemini) as tertiary fallback.

R-06 · Financial — Coin Grant Loop Exploit
Impact: High · Likelihood: Low (if idempotency enforced)
Phase 1

The POST /api/vault/grant endpoint is rate-limited (5 req/min) but lacks an idempotency check. If a user triggers it multiple times within the window before rate-limit kicks in, they could receive multiple grants.

Mitigation:

Add last_grant_at check: only grant if last_grant_at < start_of_current_month. Update coin_wallets.last_grant_at and next_grant_at atomically. Log all grant events in grant_run_log with user_id + month key UNIQUE constraint.

R-07 · JWT Secret Default Value in Production
Impact: High · Likelihood: Medium (easy to overlook)
Immediate

Auth middleware falls back to 'deploy-secret-key-change-in-production' if JWT_SECRET env var is not set. If this default value is used in production, any attacker knowing the source code can forge tokens for any user.

Mitigation:

Remove fallback default entirely. Throw error if JWT_SECRET missing. Run wrangler secret put JWT_SECRET in all environments. Add startup check in health endpoint.

R-08 · Deployment URL Spoofing (current simulation)
Impact: High · Likelihood: Certain until Phase 1 complete
Phase 1 Must-Fix

Generated deployment URLs (deploy-name-xxxxx.pages.dev) don't exist. Users clicking these links get 404s. This is the most user-visible trust issue on the platform.

Mitigation:

Implement real CF Pages API (P1-2). In interim: hide deployment URL in UI until Pages API is integrated. Show "Preparing deployment" state instead of fake URL. Never show a URL that doesn't work.

🔵 Medium
R-09 · Password Reset Token Accumulation

Stale tokens in password_resets could be used if they never expire in practice. Add cleanup to health check (DELETE WHERE expires_at < now). Enforce 1-hour expiry in reset-consumption route.

R-10 · Rate Limit Bypass via IP Rotation

Current rate limit key is IP address. VPN/proxy rotation bypasses this for auth endpoints. Mitigation: add email-based rate limiting for auth_signup and auth_forgot as secondary key. D1 table supports composite keys.

R-11 · model_usage_events vs intent_log Duplication

Both tables are now written for AI events. Causes double storage and potential confusion. Phase 2: deprecate model_usage_events writes, migrate to reading from intent_log only. Keep table for backward compat during transition.

R-12 · Orphaned Coin Holds from Network Failures

If a build request is received, coins held, then the queue message fails to send — the hold is never released. Mitigation: health check already cleans holds >10 min. Add explicit DLQ handler that releases holds and refunds coins.

R-13 · CORS Origin Whitelist Too Narrow

CORS allows only localhost:3000 and localhost:5173. Production domain (deploy-app.pages.dev) is not in the list. Will cause all API calls to fail in production. Fix: add APP_URL to allowed origins dynamically from env var.

R-14 · sessions Table Unused but Present

The platform is stateless JWT-based. The sessions table accumulates no data but adds confusion. If ever accidentally used, could create inconsistent auth behaviour. Mark as deprecated in schema comments; schedule for removal in Phase 3.

🟢 Low
R-15 · Tailwind CDN in Production

Using CDN Tailwind with custom config. Fine for MVP; for performance and CSP compliance, switch to PostCSS build in Phase 2. Also removes the "Don't use CDN in production" browser warning.

R-16 · Password Fields Not in <form>

Browser warning: password inputs outside <form> elements. Causes password manager integration issues. Wrap auth modal inputs in proper <form> elements with autocomplete attributes.

R-17 · Page Load Time 8–9.5 seconds

Page load measured at 8.64–9.5s in browser tests. Primarily CDN cold start + Tailwind CDN + FontAwesome. After migrating to build-time CSS, reduce to <2s. Add loading skeleton screens for better perceived performance.

R-18 · Admin Route Not Behind IP Allowlist

Admin panel is accessible from any IP with valid admin JWT. Consider adding Cloudflare Access (free tier) to /api/admin/* and /admin/* for an extra layer of protection.

Architecture Summary

Complete description of the updated DEPLOY Platform architecture: Intent Layer, Vault/Financial Center, Model Switcher, Home Command Center, Stripe integration, and E2E deployment flow.

Intent Layer
AI Orchestration & Audit Trail

Every AI interaction flows through a unified Intent Layer that enforces: intent classification, provider selection, fallback routing, coin charging, and immutable audit logging. No AI call is made without an Intent Layer pass.

Entry point AIService.processIntent()
Intent types complete_field, gen_spec, gen_build, revision, analyze, chat, summarize
Provider routing Primary → Fallback (OpenAI↔Anthropic)
Audit storage intent_log (SHA-256 input hash)
Coin charge Math.ceil(base_cost × multiplier)
Vault / Financial Control Center
Coin Economy + Stripe Billing

All financial operations flow through the Vault service. Coins are the platform currency; Stripe handles real money. The two systems are bridged only at webhook-confirmed payment events.

Coin operations credit / debit / hold / release / grant
Ledger coin_ledger_entries (immutable)
Hold system coins_held per build job, auto-release on failure
Analytics coin_analytics (monthly aggregates)
Revenue billing_events (idempotent, Stripe-confirmed)
Intent log GET /api/vault/intent-log
Model Switcher
Plan-Gated AI Model Selection

Users select their AI model per project. Plan level gates access to higher-capability models. The selected model is stored in projects.active_model_id and used by the Intent Layer for every AI call on that project.

Free tier gpt-4o-mini only
Member tier + gpt-4o, claude-3-5-haiku
Coin cost base_cost × multiplier (per model)
Fallback chain Selected → FALLBACK_MODELS map → Demo
Switch API PUT /api/models/select
