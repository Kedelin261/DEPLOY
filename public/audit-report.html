1B · Critical Issues & Broken Workflows

CRIT
No Automated Monthly Coin Grants

The /api/vault/grant endpoint exists but is never triggered automatically. Users must manually hit the endpoint to receive their monthly plan allocation. Cloudflare Workers Cron Trigger is not configured in wrangler.jsonc.

Fix: Add [triggers] crons = ["0 0 1 * *"] + scheduled handler in index.tsx.
CRIT
Deployment Is Simulated (No Real CF Pages API)

simulateDeployment() in deployments.ts sets a 2-second timeout then marks the record "live" with a fake .pages.dev URL. No Cloudflare Pages API call is made. Coins are debited for a fake deployment.

Fix: Implement real CF Pages API (deploy via R2 upload + Pages deploy endpoint).
CRIT
Password Reset Token Expiry Not Enforced

The forgot-password route stores a token with expiry timestamp in password_resets, but the /auth/reset-password handler does not check expires_at. Tokens are valid indefinitely.

Fix: Add AND expires_at > CURRENT_TIMESTAMP to the reset query.
CRIT
Build Notifications Never Auto-Created

When a build job completes or fails, no record is inserted into notifications for the user. The notification badge shows 0 always unless manually inserted via admin. Users have no in-app awareness of async job results.

Fix: Insert notification row in build completion/failure code paths in projects.ts.
WARN
CORS Allows Only Localhost Origins

The CORS config in index.tsx hardcodes localhost:3000 and localhost:5173 as allowed origins. Production deployment to deploy-app.pages.dev will fail cross-origin requests from the actual domain.

Fix: Add APP_URL to allowed origins or use origin wildcard for Cloudflare Pages.
WARN
Spec Transformer Has No Persistent Table

POST /api/projects/:id/transform generates a transformed spec via AI but has no dedicated transform_outputs table. Results are only returned in the response — not stored for retrieval or versioning.

Fix: Add transform_outputs table (migration 0007) and INSERT in transform handler.
WARN
Kanban Has No Backend Persistence

The Planning Kanban UI (renderKanban()) renders client-side only. There is no API route or DB table to persist tasks, columns, or card state across sessions.

Fix: Create planning_tasks table + CRUD API in Phase 2.
WARN
Deployment Rollback Is DB-Only

The rollback endpoint sets status='rolled_back' in the DB but makes no Cloudflare Pages Deployment API call. The live URL still serves the current build.

Fix: Implement CF Pages Deployments API rollback (Phase 3 deliverable).

Section 4 — Risk Areas & Mitigation Strategies

Identified per phase with likelihood, impact, and concrete mitigations

High Risk
Medium Risk
Low Risk
HIGH R1 — Coin Double-Spend / Race Condition
Phase 1 · Financial

The hold→spend flow in CoinService uses sequential DB statements, not atomic transactions. Under concurrent builds for the same user, two simultaneous holds could both succeed, spending more coins than the wallet balance allows. D1 supports BEGIN TRANSACTION.

Likelihood

Low-medium in current single-user builds, but high for high-traffic accounts with multiple concurrent builds.

Mitigation

Wrap hold + balance-check in a single BEGIN IMMEDIATE TRANSACTION. Add CHECK(balance >= 0) constraint to coin_wallets as DB-level guard.

HIGH R2 — AI Provider Outage Cascading to Build Failures
All Phases · AI

Both primary (OpenAI) and fallback (Anthropic) AI providers can be unavailable simultaneously. The demo fallback returns mocked data — users may not realize their build used placeholder content, leading to confusion and refund requests.

Likelihood

OpenAI has ~99.9% SLA; simultaneous dual outage ~0.01% but combined provider dependency creates single point of failure for AI features.

Mitigation

Return clear 503 AI Unavailable (not demo data) when all providers fail. Add Google Gemini as third fallback. Never debit coins on demo-only responses. Add status page check before AI calls.

HIGH R3 — Stripe Webhook Replay / Coin Double-Grant
Phase 1 · Financial

If stripe-webhook.ts returns a 500 for a transient error, Stripe retries the event. The current idempotency check (external_id in billing_events) may not cover all event types, potentially granting coins multiple times.

Impact

Financial loss from over-issued coins; user confusion; reconciliation complexity.

Mitigation

Store Stripe event ID in billing_events with UNIQUE constraint. Check before processing ALL event types. Return 200 for already-processed events instead of 500.

MED R4 — Cloudflare Worker 10ms CPU Limit on Complex Builds
Phase 1-2 · Performance

Build processing (AI calls, spec parsing, DB writes) is done synchronously within the request handler. The free Workers plan allows 10ms CPU time; paid allows 30ms. Complex builds with large prompts may exceed these limits.

Impact

Builds silently fail with Cloudflare's internal timeout, coin holds are never released, jobs stuck in processing state.

Mitigation

Move to Workers Paid plan. Use Cloudflare Durable Objects or Queues for async build processing. Health-check auto-recovery (already implemented) is the current safety net.

MED R5 — Production D1 Not Yet Created (Placeholder ID)
Pre-Launch · DB

wrangler.jsonc contains a placeholder database_id for the D1 database. Deploying to production without updating this will cause all DB operations to fail silently or throw binding errors.

Impact

Complete platform failure in production — all API routes that touch DB will 500.

Mitigation

Run npx wrangler d1 create deploy-production, update database_id in wrangler.jsonc, and run all 6 migrations before any production deploy. Document as checklist item #1.

MED R6 — Rate Limit D1 Table Growth Unbounded
Phase 1 · Performance

The rate_limits table is cleaned up during health-check calls only. If health-check is not called frequently, the table grows unboundedly. D1 free tier has a 5GB limit; at high traffic, rate_limit rows can accumulate rapidly.

Mitigation

Add cleanupRateLimits() call to the cron handler (Phase 1). Add TTL-based index on window_start. Consider using KV instead of D1 for rate limit counters (native TTL support).

Better Pattern

Migrate rate limiting to KV storage: await env.DEPLOY_KV.put(key, count, expirationTtl) — native expiry, no cleanup needed, faster reads.

MED R7 — CF Pages Deploy API Complexity (Phase 3)
Phase 3 · Deployment

The Cloudflare Pages deployment API requires multipart form upload, special JWT scoped tokens, and has rate limits of 500 deployments/day per project. The current R2 bucket structure may not match what Pages expects for a direct upload flow.

Mitigation

Use CF Pages API's "Direct Upload" endpoint (not Git integration). Stage build artifacts in R2, then call POST /accounts/{id}/pages/projects/{name}/deployments with form-data. Use dedicated CF API token scoped to Pages Write only.

Contingency

If Pages API proves too complex, serve generated apps from R2 public buckets as static sites. Slightly reduced functionality but faster Phase 3 delivery.

LOW R8 — Token Expiry / JWT Revocation Gap
Phase 2 · Security

JWTs are 7-day stateless tokens. There's no revocation mechanism — a compromised token remains valid for up to 7 days. The sessions table exists but is not used by the auth middleware for token validation.

Mitigation

Phase 2: implement token denylist in KV (await env.DEPLOY_KV.put('revoked:'+jti, '1', expirationTtl: 604800)). Check KV on each auth middleware call. Add JTI claim to JWT payload on sign.
